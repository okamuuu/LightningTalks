今日話すこと

  • ある案件に一人でフルスタックエンジニアすることになった
  • とはいえ手が足りないので人を雇ってもらうことになった
  • コーダーは雇えるけどフロントエンドエンジニアは雇えなかった
  • 少しでも自分の負担を減らすためにコーダーに React.js を伝授したい

今日の話に登場する人

  • サーバーサイドエンジニア
  • コーダー
  • フロントエンジニア

サーバーサイドエンジニア

API のインターフェースやデータ構造を考える人。インフラを意識してコードを書く人。

コーダー

Zeplin などのデザインの指示書やスタイルガイドから HTML, CSS を書き出す人

フロントエンジニア

書き出された HTML, CSS 元に、npm, webpack を使ってライブラリを使って簡素な HTML と minify された Javascript を用意する人。

ここではフロントエンドエンジニアに Flash のようなインタラクティブデザインは求めていない

思惑

  • React を使ってサービス作りたい
  • とはいえ全ての修正対応を自分がしたくない
  • というわけで React.js を使える人材を雇いたい
  • もしくは React.js が使える人材を育てたい

結果、コーダーさんに React.js を伝授することになった

色々試してみたが...

  • webpack の設定方法がわからない
  • 設定方法をドキュメント化してみる
  • どこにファイルを作っていいかわからない
  • react-router の記述は全部こちらでやっておこう
  • DatePicker など外部ライブラリはどうやれば...
  • 難易度が高いよね。よしそれもこちらで巻き取ろう

やはりコーダーには React.js は荷が重いようだ...

残念な結果だが、自身に変化があった

frontend の何たるかが少しわかった気がする

  • 仕上がったデザイン指示書を見ながら画面遷移をイメージする
  • 噛み合わない仕様があれば早い段階で気づく
  • WEB API の interface に密接に関わる
  • API が不適切だったりすると早い段階で気づく
  • デザイン指示書の要素が欠落していたりすると早い段階で気づく

デザイナー、プランナー、エンジニアがそれぞれ思い描いている歯車が噛み合わない事に早い段階で気づく

フロントエンドエンジニアは真ん中にいる

結論

強いフロントエンドエンジニアが居たら勝てる
